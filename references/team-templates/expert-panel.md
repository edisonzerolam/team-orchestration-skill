# Expert Panel（专家小组讨论会）

## 适用场景

需要多位专业角色从不同角度分析同一组数据，通过交叉辩论形成共识。

典型场景：
- 策略效果评审（组合/风控/归因）
- 架构设计评审（性能/安全/可维护性）
- 投资决策（宏观/行业/估值）
- 代码审查（逻辑/安全/性能）
- **内容生产审读（写作/编辑/读者/事实核查/风控）** ⭐ 新增

---

## 新增：内容生产专家小组模板

基于《广钢最生活》公众号多轮实战沉淀，覆盖从选题到发布的完整链条。适用于深度报告、公众号长文、调研类内容。

---

### 一、完整生产流水线（4阶段）

```
Phase 1: 选题与调研 → Phase 2: 写作与打磨 → Phase 3: 审读发布 → Phase 4: 学习复盘
                    ↓                   ↓                   ↓              ↓
              素材+大纲              定稿+勘误           质检+配图         数据反馈
```

#### Phase 1: 选题与调研（素材准备）

| 步骤 | 操作 | 产出物 | 建议耗时 |
|------|------|--------|:--------:|
| 1.1 选题评估 | `@general` + 互联网搜索，评估选题的传播力/独家性/时效性 | 选题评分表（传播力/深度/风险 三维打分） | 1次交互 |
| 1.2 素材收集 | `@general` + `anysearch` skill，并行搜索 6-8 个维度的相关信息 | 素材文档（结构化，含来源URL+等级标记） | 1-2次 |
| 1.3 大纲撰写 | `@article-writer` 基于素材输出文章大纲 | 大纲（含每个章节的核心论点+数据点） | 1次 |
| 1.4 专家评审 | 调用下方审读流程的「内容质量组」，评估大纲方向 | 大纲评审意见 | 1次 |

**素材分级规则**：每份素材按来源可靠性标记等级：
- **P0 - 原始文件**：控规说明书、政府批文、原始PPTX/PDF
- **P1 - 官方发布**：政府官网公告、卫健委公示、规自局公示
- **P2 - 媒体转载**：新闻报道、自媒体报道（需交叉验证）
- **P3 - 作者推断**：基于P0-P2的推理分析（需显式标注）

**并行搜索模板：**
```
task:
  subagent_type: "@general"
  prompt: |
    以下为 N 个独立搜索方向，请并行执行，每个方向输出结构化结果：
    
    方向1：[XXX] — 关注最新进展、关键数据
    方向2：[XXX] — 关注历史背景、政策文件
    ...
    
    每个方向输出：
    - 核心发现（带数据引用）
    - 来源URL（可访问）
    - 关键人物/事件/时间点
    - 素材等级（P0-P3）
    汇总为一个素材文档。
```

#### Phase 2: 写作与打磨（内容产出）

| 步骤 | 操作 | 产出物 |
|------|------|--------|
| 2.1 初稿 | `@article-writer` 按大纲撰写初稿 | 初稿 |
| 2.2 时间自洽检查 | 检查所有"今年/去年/本月"是否替换为具体年份（2024、2025、2026） | 时间勘误表 |
| 2.3 事实核查 | `@general` + `anysearch`，核查初稿中所有数据claim，区分原始文件 vs 媒体 vs 推断 | 勘误表 |
| 2.4 用户审阅 | 用户提出修改意见 | 修改清单 |
| 2.5 多轮修改 | `@article-writer` 逐轮应用修改意见 | 迭代稿 |
| 2.6 最终版 | 用户确认内容 | 定稿 |

#### Phase 3: 审读发布（自动路由）

按自动路由规则执行下方审读流程。审读通过后进入发布：

```
审读通过 → platform-qc检测 → visual-planner（配图检查）→ 配图下载归档 → 交付
                                  └─ 朱雀AI < 20%?
                                  └─ 敏感词检查?
                                  └─ 平台格式?
```

#### Phase 4: 学习复盘

```
发布后72小时内：
- 记录阅读量/互动率/转化数据
- 对比选题时预估的传播力得分，更新评分模型
- 记录审读中发现的典型问题，补充到"实战经验"表
- 归档到知识库备查
```

### 二、审读流程（自动路由 + 并行审读）

#### 自动路由规则

收到定稿后，先判定审读深度：

```
输入定稿 → 判断：
  ├─ 字数 > 3000 字 + 含数据/引文 → 03-03-03（完整模式：3组×3轮×3通道）
  ├─ 字数 1000-3000 字 → 02-02-02（默认模式：2组×2轮×2通道）
  ├─ 字数 < 1000 字 → 01-01-01（速审批模式：1轮+platform-qc）
  └─ 含以下任一关键词 → 强制升级到 03-03-03：
       - 绝对化表述："唯一""最差""一个都没有"
       - 点名批评：具体企业/学校/个人名称+负面评价
       - 敏感事件：维权、冲突、官方失误
```

#### 团队组成



| 组别 | 角色 | 推荐 subagent | 审读维度 |
|------|------|--------------|---------|
| **内容质量组** | 写作逻辑 + 自然语言 | `@article-writer` + 加载 `humanizer-zh` skill | 结构逻辑、观点深度、AI味检测、去AI味修正 |
| **事实合规组** | 事实核查 + 风险控制 | `@general`（带来源交叉验证）+ 加载 `anysearch` skill | 数据准确性、数学一致性、删文/投诉风险 |

#### Round 1: 并行审读（2路同时派发）

```
task:
  subagent_type: "@article-writer"
  prompt: |
    ## 内容质量组审读
    
    请完成以下三项任务并输出：
    
    ### A. 写作逻辑审读
    - 每段内部的事实→分析→判断顺序是否合理
    - 段落间过渡是否自然，有无跳脱、重复
    - 头尾是否呼应，标题与正文是否一致
    - 数据口径是否统一（如"32所"前后是否一致）
    
    ### B. 自然语言优化
    加载 humanizer-zh 检测以下模式：
    - 三段式排比、破折号滥用、过分工整对仗
    - "找补"语句（替政府/企业解释的倾向）
    - "随着""在...背景下""值得注意的是"等AI高频词
    
    ### C. 输出修正版
    直接输出优化后的完整版本，标注改动位置。

task:
  subagent_type: "@general"
  prompt: |
    ## 事实合规组审读
    
    加载 anysearch 进行来源验证。完成以下两项任务：
    
    ### A. 事实核查
    1. 来源交叉验证：每个数据是否可追溯到原始来源？
       - 区分"原始规划文件" vs "媒体报道" vs "作者推断"
       - URL 是否可访问，页面是否包含所说的内容
       - 如有原始规划文件，优先于二手报道
    2. 数学一致性：总数=分项之和？比率计算正确？
    3. 地理/坐标验证：地名claims（地铁站、商业体、医院）
       - 有 Baidu Maps API 时用 geocoding 核实
       - 距离 claims（"距100米""3站地铁"）
    4. 新旧口径区分：规划原文 vs 实施方案
    
    输出逐条核查结果：✅ 准确 / ⚠️ 需确认 / ❌ 错误
    
    ### B. 风险控制
    - 有无触发微信删文的表述？（群体事件描述、煽动情绪）
    - 有无指名贬低特定学校/企业/个人？（"排不上号"等）
    - 有无可能被投诉的数据争议？（"唯一没有""全市最差"）
    - 有无超过批评边界？（区分"事实陈述"和"价值判断"）
    
    输出风险等级：🔴 高危 / 🟡 中危 / 🟢 低危
    输出勘误表 + 风险处置建议。
```

#### Round 2: 终审

```
通道1：platform-qc 检测
  ├─ 朱雀AI分数 < 20%？
  ├─ 平台格式合规？
  └─ 敏感词检查？

通道2：visual-planner（可选）
  └─ 检查关键论点是否有配图
  └─ 封面图比例、数据可视化建议

输出：🟢 PASS / 🟡 CONDITIONAL / 🔴 FAIL

**否决权规则**：
- 事实合规组标 🔴 高危（如数据造假、法律风险）→ **一票否决**，退回重写
- 内容质量组标 🔴 严重写作问题 → 退回修改，不阻塞发布但需附修改说明
- 两组分歧（一组PASS一组FAIL）→ 升级到主 agent 裁决，必要时走03-03-03完整模式
```

### 03-03-03 完整模式（数据密集型报告扩展）

**02-02-02** = 2组（质量+合规）× 2轮（审读+终审）× 2通道（质检+配图）——默认
**03-03-03** = 3组（质量+合规+领域）× 3轮（审读+裁决+终审）× 3通道（质检+配图+第三方验证）——完整

当自动路由判定为完整模式时，在02-02-02基础上增加：

1. Round 1 分拆为3路：「内容质量组」+「事实合规组」+「领域专家组」（`@general` 加载领域知识提示词，如教育/医疗/交通等）
2. Round 2 增加矛盾裁决轮（当两组分歧时）
3. 终审六维评分（行为红线、内容质量、写作质量、逻辑一致性、合规、完整性）

## 实战经验：内容生产专家小组常见问题

基于多轮审读实战沉淀：

### 写作层面

| 问题 | 识别方法 | 修复方案 |
|------|---------|---------|
| 找补语气 | "思路是对的""两边都有道理""走快有原因、走慢有难处" | 删掉平衡句式，让事实自己说话 |
| 数据堆砌 | 连续3行以上纯数据，缺分析和判断 | 先问"这个数据说明了什么？"再写下一句 |
| 前后口径不一致 | "规划32所"后面变成"规划2所初中" | 全文统一引用维度（32所=15幼+9小+5初+3高） |
| 公式不自洽 | 32-4-9-10=9 vs 只说32-10=22 | 逐项核查数学关系，确保总数=分项之和 |

### 事实核查层面

| 问题 | 识别方法 | 修复方案 |
|------|---------|---------|
| 二手数据偏差 | 媒体报道口径vs控规原文不一致 | 优先查原始规划文件/政府报告，非媒体报道 |
| 坐标/距离偏差 | "距100米""3站地铁"无验证 | 用百度地图API geocoding核实 |
| 时间线错误 | "今年"/"去年"在跨年发布时失效 | 全部替换为具体年份（2024年、2025年） |
| 新旧口径混淆 | 2025年实施方案vs2013年控规 | 标注数据对应的时间节点 |

### 风险控制层面

| 问题 | 识别方法 | 修复方案 |
|------|---------|---------|
| 贬低具体学校/企业 | "排不上号""最差""没人去" | 改为"不是优先选择范围"/"不在第一梯队" |
| 群体事件描述 | "拉横幅""万人签名""请愿" | 改为"通过12345反映诉求"/"集中表达意见" |
| 绝对化表述 | "唯一没有""全市最差""一个都没" | 加上限定词"广州重点新城中" |
| 企业点名批评 | "XX公司能拖就拖" | 改为行业通用描述，不点名 |

## 原有的模板（投资分析方向）

### 团队组成

| 角色 | 推荐 subagent | 专业视角 |
|------|--------------|---------|
| 组合/策略分析师 | `@quant-researcher` | 因子有效性、持仓集中度、策略逻辑 |
| 风控/交易质量 | `@risk-manager` | 合规违规、TP/SL有效性、集中度风险 |
| 收益归因 | `@debug-expert` 或 `@quant-researcher` | PnL分解、费用效率、尾部风险 |
| 其他可选 | `@code-reviewer` 等 | 按需扩展 |

### 完整工作流

#### Phase 1: 独立分析（并行派发）

主 agent 给每个专家派发 task：

```
task:
  subagent_type: "@quant-researcher"  # 或其他专家
  prompt: |
    ## 任务: [主题] - [专家角色]分析
    
    ### 数据源
    [列出需要分析的所有文件/数据]
    
    ### 交付要求
    1. 从你的专业角度（组合结构）分析数据
    2. 输出结构化报告，包含：
       - 核心发现（带数据引用）
       - 风险标识（L1-L4严重度）
       - 优先级建议（P0/P1/P2）
    3. 保存到: [路径]/expert_[角色]_report.md
```

#### Phase 2: 交叉审阅（串行）

所有专家完成后，主 agent 收集报告，发给每个专家审阅其他人的：

```
task:
  subagent_type: "@risk-manager"
  prompt: |
    ## 任务: 交叉审阅 Expert 1 的报告
    
    ### 待审阅报告
    [Expert 1 的完整报告内容]
    
    ### 要求
    1. 从你的专业角度（风控）质疑 Expert 1 的发现
    2. 对每个分歧点标注：
       - ✅ 同意
       - ⚠️ 有疑问（说明理由）
       - ❌ 不同意（说明理由+数据）
    3. 输出到: [路径]/review_[角色]_of_[被审人].md
```

#### Phase 3: 小组辩论（主 agent 主持）

主 agent 汇总所有分歧点和共识点，逐议题主持辩论：

```
## 议题 N: [标题]

### Expert X 的主张:
[引用数据和结论]

### Expert Y 的反驳:
[引用数据和结论]

### Expert Z 的补充:
[引用数据和结论]

### 裁决:
✅ 推荐方案: [具体方案]
📊 依据: [关键数据]
❌ 反对意见: [如有，记录]
⚡ 实施成本: [低/中/高]
```

辩论规则：
1. 每个议题必须至少包含 "主张→反驳→回应" 三轮
2. 无法达成一致时，记录多数派和少数派观点
3. 所有结论必须引用具体数据行号或指标值

#### Phase 4: 最终方案

输出优先级排序的实施方案：

| 优先级 | 议题 | 推荐方案 | 数据依据 | 实施成本 |
|--------|------|---------|---------|---------|
| P0 | ... | ... | ... | 低 |

附：少数派声明（如有）

## 通用常见陷阱

| 陷阱 | 表现 | 预防 |
|------|------|------|
| 主 agent 代劳 | 主 agent 用自己的话总结专家观点 | 严格执行 R6，必须由 subagent 亲自输出 |
| 角色混淆 | 风控分析师讨论因子权重 | 派发时明确每个角色的视角边界 |
| 数据不足 | 专家基于不完整数据下结论 | Phase 1 前确认每个专家的数据源充分 |
| 回避冲突 | 分歧被模糊处理成"都有道理" | 必须记录分歧，至少一轮反驳 |
| 样本太小 | 基于少量事件做重大决策 | 标注结论的统计显著性（如"基于14次事件"） |
| **来源未交叉验证** | 只查报道不查原始规划文件 | 内容生产场景必须同时查原始文件+报道+地图 | ⭐ 新增 |
| **数学未自洽** | 前后数字口径不一致 | 每次修改后重新计算所有比率和总数 | ⭐ 新增 |
