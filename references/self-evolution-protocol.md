# 三环自进化协议（v2.3 增强版）

基于 Loop Engineering 深度调研成果（`references/loop-engineering-research.md`），
融合控制理论、反馈循环六级分类学、生产级自进化系统最佳实践。

---

## 架构总览：三层循环 + 三个维度

```
时间尺度:   L1 执行层(秒级)    L2 策略层(分钟级)   L3 元学习层(小时/天级)
对应 Loop:  Loop 1            Loop 2              Loop 3
           执行中反思          事后回顾             主动联网增强
           
三个监控维度贯穿所有层级:
  ┌────────────────────────────────────────────────────────┐
  │  ① 增益监控 (LoopGain): 实时检测 Aβ = E(n)/E(n-1)    │
  │  ② 反馈质量工程: 及时性/准确性/一致性/特异性/可操     │
  │  ③ 退化防护: 上下文预算 + 周期重置 + 状态外部化       │
  └────────────────────────────────────────────────────────┘
```

### 六种反馈信号类型（Team Orchestration 映射）

| 信号 | 理论分类 | 增益机制 | 阻尼机制 |
|------|---------|---------|---------|
| 专家匹配失败 | 负反馈/行为反馈 | 连续3次→冷却(降权) | 指数平滑衰减(α=0.3) |
| 用户明确认可 | 正反馈/社交反馈 | 加深方向探索 | 上限封顶(防过调) |
| 用户提出修改 | 负反馈/行为反馈 | 记录偏误→不重复 | 仅会话内生效(窗口化) |
| 交叉验证冲突 | 负反馈/信息反馈 | 提升验证深度级别 | 逐级升级(不跳级) |
| 审核门 FAIL | 负反馈/结果反馈 | 退回重做(≤3轮) | 第3轮→争议报告 |
| 无反馈/超时 | 无反馈 | 熔断/回滚 | 自动切换到备用路径 |

## Loop 1: 执行中反思（L1 执行层）

**触发条件**：每个 subagent 执行完毕后自动触发

**增强（v2.3）**：
1. **增益监控**：计算 Aβ = 当前误差/上轮误差
   - Aβ < 1 → 继续（收敛）
   - Aβ ≈ 1 → 降低权重（振荡风险）
   - Aβ > 1 → 熔断+回滚到最佳输出
2. **上下文预算**：token 利用率 > 80% 时自动摘要旧记录
3. **反馈质量自检**：对反思内容做六维评分

**行为**：
```markdown
## 执行反思 (追加到 self-evolution-log.md)

- 任务: {任务简述}
- 时间: {时间戳}
- 有效方法: {什么方法/提示词效果好}
- 缺失数据: {什么数据获取不到 / 质量差}
- 改进建议: {下次如何优化}
- 环路增益: {Aβ 值} (收敛/振荡/发散)
- 上下文预算: {已用/总窗口 tokens}
```

**数据流**：只写入本专家目录下的 `self-evolution-log.md`

**防退化**：每 N 次子任务执行 → 新会话/重置上下文

## Loop 2: 事后回顾（L2 策略层）

**触发条件**：整个多专家任务完成后

**增强（v2.3）**：

1. **Polyak 平均更新 expert-scores**：
   ```python
   new_score = τ * old_score + (1-τ) * this_round_score
   # τ=0.9 (稳定) / τ=0.7 (快速适应)
   ```

2. **多轮退化检测**：
   - 分析当前任务与最近 N 个任务的 expert-scores 趋势
   - 连续下降 3 次 → 标记该专家退化，触发 Loop 3 重新校准

3. **收敛速度记录**：
   - 记录从开始到完成的循环次数
   - 记录每次修正后的错误减少比率
   - 用于调整下次的 `max_iterations` 预算

**行为**：
1. 收集所有参与专家的 self-evolution-log（含增益数据）
2. 横评各专家表现（质量、速度、数据完整度、振荡频率）
3. Polyak 平均更新 `expert-scores.json`
4. 生成跨专家学习建议（含退化警告）

**输出**：更新后的评分数据库 + 退化报告

## Loop 3: 主动联网增强（L3 元学习层）

**触发条件**：定时（每 N 次执行）或手动触发

**增强（v2.3）**：

1. **OODA 定向增强**：
   - 维护一个持久世界模型（跨 session 的知识图谱）
   - 联网搜索的结果先更新世界模型，再更新知识文件
   - 世界模型作为所有 Agent 的共享上下文（而非每次重新搜索）

2. **双重进化通道**：
   - **快进化**：发现过时信息/新工具 → 直接更新 knowledge 文件
   - **慢进化**：发现跨 session 的模式 → 更新 expert-matching 权重

3. **验证器过滤**：
   - 联网搜索结果经验证器审查后写入（防止递归污染）
   - 一致性 < 0.3 的搜索发现标记「低置信」，暂不合并

**行为**：
1. 遍历所有 active 专家
2. 对每个专家按领域关键词搜索互联网
3. 生成进化报告:
   - 新发现的知识点
   - 已过时的信息
   - 可用的新工具/API
   - 元学习趋势（跨 session 的改进方向）
4. 用户审核后:
   - 通过 → `knowledge-merger.py` 合并到 knowledge 文件
   - 拒绝 → 丢弃（写入 discard-log 供事后分析）

**数据流**：半自动 — 用户确认后写入

## 附加：反馈信号质量工程

所有循环产出的反馈信号，在执行前经过六维质量自检：

| 维度 | 检查项 | FAIL 处理 |
|------|--------|----------|
| 及时性 | 反馈是否在超时前到达 | 标记为延迟信号，降权 |
| 准确性 | 是否经交叉验证确认 | 未验证 → 降级为 medium |
| 一致性 | 温度=0 评估，无误报 | 浮动大 → 重置再测 |
| 特异性 | 是否指出具体问题位置 | 模糊反馈 → 丢弃 |
| 可操作性 | 是否包含具体修复建议 | 无建议 → 标记为参考 |
| 安全性 | 是否引入新风险 | 负面 → 丢弃+红队记录 |

## 附加：熔断与回滚机制

任何 Loop 触发以下条件之一即熔断：

| 条件 | 检测方法 | 动作 |
|------|---------|------|
| Aβ > 1.5 连续 2 次 | LoopGain | 立即中止，回滚最佳输出 |
| token 利用率 > 95% | 上下文预算监控 | 裁剪至 70% |
| expert-score 连续下降 3 次 | 趋势分析 | 冷却该专家，触发 Loop 3 |
| 循环次数 > `max_iterations` | 计数器 | 输出当前最佳结果 |
| 用户连续 3 次纠正同一问题 | 偏误累计 | 停止当前 Loop，标记为 Pattern |
